# v0.1.0 Release Notes

## What This Release Is

v0.1.0 positions repo-health-doctor as a local-first pre-execution safety gate
and evidence normalizer for AI agents and developers reviewing unfamiliar
repositories.

It collects bounded evidence, records limitations, and keeps missing or degraded
visibility from becoming false confidence before repository-derived commands are
run.

Use it before an AI agent or developer runs commands from a repository they do
not trust yet.

## What This Release Is Not

- It is not a replacement for Gitleaks, TruffleHog, OSV-Scanner, Grype, Syft,
  zizmor, actionlint, Falco, Tracee, or an EDR.
- It is not a complete malware sandbox.
- It does not prove that a repository is safe.
- It does not authorize live execution from a gate decision.
- It does not claim third-party security review has happened.

## Stable Contracts

- Default v3 JSON output compatibility remains stable.
- Default CLI behavior remains stable.
- Redaction principles remain stable.
- No finding is not proof of safety.
- Gate decisions and execution authorization remain separate.
- Gate decisions keep `execution_authorized=false`.
- Limitations are gate inputs, not decorative notes.

See [../public-contracts.md](../public-contracts.md).

## Experimental Surfaces

- Evidence schema draft.
- Gate decision schema draft.
- Opt-in `--gate-decision-output`.
- Plan-only AI agent preflight demo script. It does not change global hook
  configuration and does not execute the displayed target command.
- C-0 field research safety protocol. It is a docs-only protocol for future
  private review and publication decisions, not target collection or public
  naming automation.
- Imported Gitleaks and OSV-Scanner evidence adapters. They are evidence import
  paths, not scanner replacements.
- Real Gitleaks, OSV-Scanner, and Trivy scanner adapters. They are explicit
  adapter/API surfaces, not default CLI scanner execution.
- Sample outputs.
- Execution authorization artifact and validator.
- Docker `sandbox-run` v1 core runtime, including gate / authorization binding,
  local-image-only Docker mode, deterministic Docker argv, disposable workspace
  copy, copy-budget fail-closed behavior, bounded redacted output previews, and
  the draft `schemas/sandbox-run.schema.json` report.
- Real-output-compatible fixture coverage.
- Docker integration CI path.

Experimental surfaces may change during the v0.x series. See
[../versioning.md](../versioning.md).

## Known Limitations

- Third-party security review is not done.
- Real compatibility is limited to documented fixture, version, and scope.
- Docker boundary tests in normal CI use fake/local paths.
- Optional real Docker sandbox-run execution depends on the runner environment
  and an already-present local image; repo-health-doctor does not pull images
  automatically.
- A successful sandbox-run is bounded execution evidence only, not a safety
  proof and not unrestricted execution authorization.
- Successful sandbox-run does not mean safe and does not mean authorization to
  continue.
- External adapters remain experimental evidence import paths.
- Current default v3 JSON is check-oriented, not the final formal evidence/gate
  model.

## Third-Party Review Status

Third-party security review is not done and remains external required work.

Use the security model review issue template when requesting external review:

## Upgrade And Compatibility Note

The initial public release preserves the default v3 JSON report and default CLI
behavior. Opt-in sidecars and experimental schemas are intentionally separate
from that stable path.
