# Controlled Single-Command Live Gate Design

## Status And Boundary

This document designs a future controlled single-command live gate. It does
not implement Docker inspect, Docker pull, Docker run, strace, runtime hooks,
observer capture, Phase 1/2/3 live execution, approval promotion, or unknown
repository command execution.

The live gate is the first future point where a command could be attempted,
but only after a separate implementation and explicit human authorization. A
non-executing preflight PASS remains a prerequisite, not an execution permit.

## Live Gate Responsibility

The live gate is responsible for one exact command only. It must revalidate
the supplied human-created approval artifact, exact candidate key, image lock,
behavior policy binding, command identity, disposable workspace, runtime
safety flags, observer capture readiness, normalized evidence, and behavior
policy verdict.

The gate must preserve these boundaries:

- `shell: false` and `network_policy: none`;
- no writable mount of the original host repository;
- no host HOME, credentials, or Docker socket mount;
- disposable workspace and sandbox-owned writable tmp/cache/home paths only;
- observer capture enabled before command execution;
- normalized evidence generated before final PASS consideration;
- behavior policy verdict applied after execution evidence is validated.

Evidence missing, observer degradation, unavailable observers, missing strace
logs, parse failure, invalid normalized evidence, or behavior-policy BLOCK
must prevent final PASS.

## Transition From Non-Executing Preflight

The live gate may be entered only when all of the following hold:

- `sandbox_runner_preflight` returned `verdict: pass`;
- approval artifact validation passed and the artifact is not expired or
  revoked;
- image lock validation and image lock-binding validation passed;
- behavior policy validation and behavior-policy binding validation passed;
- normalized observer evidence schema support is available for the future
  capture output;
- static transition validation passed;
- repository identity, reviewed commit, and clean working tree match the
  approval;
- candidate key and exact-match key match the approval and preflight inputs;
- image digest or full image ID matches the image lock and approval binding;
- behavior policy ID and fingerprint match the approval binding;
- phase, kind, cwd, argv, environment allowlist, shell setting, and network
  policy match exactly;
- the risk tier is not T4 or T5;
- T3 has complete exception metadata if a future controlled exception allows
  it.

The live gate must repeat critical checks instead of trusting the preflight
report alone.

## Approval Artifact Handling

An approval artifact is command-scoped evidence of a human review decision. It
is not generated by the live gate and is not derived from an approval draft.
Finding a valid approval artifact does not by itself authorize execution.

The live gate must reject an approval when it is expired, revoked, malformed,
dirty-worktree-bound, commit-mismatched, candidate-mismatched, phase-mismatched,
or missing required reviewer and lifecycle metadata. Phase 2 and Phase 3
approvals remain separate artifacts and cannot authorize each other.

## Docker Image Attestation

Image setup and command execution are separate phases. A human-operated setup
workflow may pull a reviewed digest and record an image lock outside the live
gate. Runtime execution must use `--pull=never`.

A future Docker inspect step must compare the already-present image against
the image lock and approval binding:

- registry digest for registry-primary images;
- full image ID for explicitly sanctioned local development images;
- platform and tool-version inventory;
- image lock ID and distribution type;
- Docker Desktop, Docker Engine, runc, and containerd update attestation
  supplied by the operator.

Docker inspect failure, ambiguous results, digest mismatch, image ID mismatch,
platform mismatch, tool mismatch, or missing operator runtime attestation must
fail closed. Image lock rotation invalidates approvals bound to the previous
lock, digest, or full image ID. Local sanctioned images remain development-only
and portability-limited.

The future image attestation report should include only safe identifiers,
boolean match results, tool-version categories, limitation labels, and
redaction status. It must not include raw host paths or credentials.

The static image-attestation skeleton is implemented as
`sandbox_image_attestation`. It validates supplied attestation data and image
lock consistency only. It is not Docker inspect and does not authorize
execution; a future live gate must still obtain real image facts from a
separate Docker inspection phase.

## Docker Runtime Safety Flags

Future live execution must construct Docker argv from a closed plan, not from
shell text. Required runtime constraints are:

- `--pull=never`;
- `--network none`;
- `--cap-drop ALL`;
- `--security-opt no-new-privileges`;
- read-only container root filesystem;
- explicit non-root user;
- pids, memory, and CPU limits;
- explicit entrypoint;
- argv array only, with `shell: false`;
- no Docker socket mount;
- no host HOME or credential mount;
- disposable workspace mount only;
- sandbox-owned writable tmp/cache/home paths only.

The implementation should also evaluate a seccomp profile, tmpfs usage,
read-only binds, user namespace isolation, strict environment allowlists, cwd
validation, and mount redaction. Any omitted hardening must be recorded as a
limitation in the report.

## Disposable Workspace And Mount Policy

The original repository must not be writable from the container. The runner
must materialize a disposable workspace for the reviewed commit and mount only
that workspace plus sandbox-owned writable tmp/cache/home paths.

Unsafe symlinks, skipped files, secret-like files, and files excluded from the
materialized workspace must be recorded by safe category and repo-relative
handle only. Raw host paths must be redacted.

Logical writable paths are limited to:

- `/workspace` for the disposable copy when writes are explicitly allowed by
  policy;
- `/tmp/tmp`;
- `/tmp/home`;
- `/tmp/pip-cache`;
- `/tmp/npm-cache`;
- `/tmp/xdg-cache`.

Cleanup must be reported. Cleanup failure, outside writable delete, denied
read, host HOME access, Docker socket access, or write outside allowed prefixes
prevents final PASS.

## Observer Capture Contract

The controlled live gate should use strace plus runtime hook capture when the
policy requires both. Capture must start before the command and produce a
normalized observer evidence document after the command exits or times out.

The report must never include raw syscall log lines, raw stdout, raw stderr,
host paths, credentials, or unredacted observer errors. Raw evidence may exist
only in a disposable, access-controlled evidence location with a safe handle.

Normalized evidence must use the existing
`sandbox_normalized_observer_evidence` contract and include safe evidence ID or
fingerprint, source flags, command fingerprint, execution outcome, event
counts, flags, summaries, limitations, residual risks, and redaction status.

Observer unavailable, strace log missing, strace parse failure, runtime hook
unavailable or inactive when required, runtime hook parse failure,
`observer_degraded: true`, or `evidence_complete: false` prevents final PASS.

## Behavior Policy Verdict

The behavior policy evaluator consumes normalized evidence after the live
command attempt. The final live result must apply the verdict conservatively:

- policy `block` means final BLOCK;
- policy `warn` or `needs_review` is not a successful live result;
- policy `pass` is required, but still does not prove repository safety beyond
  the observed command scope.

Timeout, return-code mismatch, unexpected exec, subprocess spawn, network
event, outside write, secret event, denied read, Docker socket access, host
HOME access, and outside writable delete must be reflected in normalized
evidence and policy evaluation. Existing default-deny blockers must not be
relaxed by the live gate.

## Controlled Fixture Versus Unknown Repository

The first live implementation must be controlled-fixture-only. Controlled
fixtures may use repository-owned static approvals, images, policies, and
expected command identities created solely for test validation.

Unknown repository live execution remains disabled until a separate design
review and explicit maintainer authorization. T1, T2, and T3 unknown
repositories are not live-eligible merely because static gates pass. T4 and T5
remain ineligible.

Controlled fixture live execution should prove the gate mechanics, evidence
normalization, policy evaluation, redaction, and cleanup behavior before any
unknown repository rollout is considered.

## Live Gate Report

The proposed report uses `schema_version: "0.1-draft"` and
`report_kind: "sandbox_single_command_live_gate"`.

Required fields:

- `mode`;
- `execution_permitted`;
- `execution_performed`;
- `docker_contacted`;
- `docker_run_performed`;
- `observer_capture_performed`;
- `phase_live_performed`;
- `approval_reference`;
- `preflight_reference`;
- `image_attestation`;
- `command_identity`;
- `runtime_safety_flags`;
- `disposable_workspace`;
- `observer_summary`;
- `normalized_evidence_reference`;
- `behavior_policy_verdict`;
- `final_verdict`;
- `blockers`;
- `warnings`;
- `limitations`;
- `residual_risks`;
- `cleanup`;
- `redaction_status`.

The report separates `execution_permitted` from `execution_performed` so a
blocked gate can explain that no command ran. Raw logs, stdout, stderr, host
paths, credentials, and raw policy values are excluded.

## Fail-Closed Conditions

The live gate must BLOCK for:

- preflight not PASS;
- approval invalid, expired, or revoked;
- candidate key, exact-match key, commit, repository identity, or command
  mismatch;
- dirty working tree;
- T4/T5 or incomplete T3 exception metadata;
- image lock mismatch;
- Docker inspect failure or mismatch;
- Docker run failure before trustworthy evidence exists;
- observer unavailable, missing strace log, parse failure, or degraded
  observer;
- normalized evidence invalid or incomplete;
- behavior policy binding mismatch;
- behavior policy verdict BLOCK;
- network event, secret event, Docker socket access, host HOME access, denied
  read, outside write, or outside writable delete;
- raw log, stdout, stderr, host path, or secret-like value leak attempt;
- cleanup failure;
- unknown safety-relevant field.

## Implementation Split

Recommended commit sequence:

1. Add this live gate design and report schema only; no runtime code.
2. Add image attestation static report skeleton; no Docker inspect.
3. Add Docker command builder dry-run report; no Docker execution.
4. Add disposable workspace live-readiness verifier; no command execution.
5. Add observer capture planner; no strace or runtime-hook execution.
6. Add non-executing live gate report aggregator.
7. Add controlled-fixture live gate behind an explicit flag after human
   approval.
8. Add controlled-fixture live smoke tests after human approval.
9. Revisit unknown repository live gate eligibility in a separate design
   review.
